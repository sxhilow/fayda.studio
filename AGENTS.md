# AGENTS.md — Fayda Studio

Marketing/portfolio site for Fayda.studio. Static, Astro-driven, single repo, no backend.

## Stack

- Astro 6 (`^6.1.3`) with `<ClientRouter />` view transitions
- Tailwind CSS v4 via the `@tailwindcss/vite` plugin (config lives in `src/styles/global.css` via `@theme` — there is no `tailwind.config.*`)
- MDX via `@astrojs/mdx`; `astro-sitemap` for sitemap output
- PostHog (`posthog-js`) and `@vercel/analytics/astro` for analytics
- `lenis` for smooth scroll, wired in `src/layouts/Layout.astro`
- TypeScript strict (`extends: "astro/tsconfigs/strict"`)

## Commands

All run from the repo root.

| Command | What it does |
| --- | --- |
| `npm install` | Install deps (lockfile is `package-lock.json`) |
| `npm run dev` | Dev server on `http://localhost:4321` |
| `npm run build` | Static build to `./dist/` |
| `npm run preview` | Serve the built `dist/` locally |
| `npm run type` | Runs `astro check` (the typecheck — not raw `tsc`) |
| `npm run astro -- ...` | Pass-through to the Astro CLI (e.g. `astro add`, `astro check --watch`) |

There is **no test runner, linter, or formatter wired into npm scripts**. `prettier` is present in `node_modules` (transitive), but no `format`/`lint` script exists. To verify a change locally use `npm run type` then `npm run build`. There is no CI workflow (`.github/workflows/` is absent).

## Repo layout (only what matters)

- `src/pages/` — routes. Every page must wrap its content in `<Layout>` from `src/layouts/Layout.astro`. There is exactly **one** layout.
- `src/layouts/Layout.astro` — the only layout. Owns `<head>` (title, description, favicon, Google site verification meta, `<Seo>`, `<ClientRouter />`, `<PostHog />`), sticky `<Header>` chrome, `<main><slot/></main>`, `<Footer>`, `<Analytics />`, and the Lenis scroll init script. The `<html>` tag uses `transition:name="root" transition:animate="none"` — do not remove.
- `src/components/`
  - `chrome/` — `Header.astro`, `Footer.astro`
  - `ui/` — reusable primitives (Button, Card variants, Pill, Section wrapper, etc.)
  - `sections/` — page sections. `home/` is home-page-only, `case-study/` is case-study-only; top-level files are shared (e.g. `Plans.astro`, `Contact.astro`, `FAQs.astro`, `BookCall.astro`).
  - `seo/` — `Seo.astro` (rendered by `Layout.astro`)
  - `integration/` — third-party snippets. `posthog.astro` lives here.
- `src/data/` — TS data modules (`plans.ts`, `faqs.ts`, `projects.ts`, `services.ts`, `why-us.ts`, `process.ts`, `whatsapp.ts`, `index.ts` for nav/footer). SEO data is at `src/data/seo/siteData.json` and `src/data/seo/jsonLD.js` — note the `seo/` subdir, not the data root.
- `src/content/case-studies/*.mdx` — case study content collection. Schema is defined in `src/content.config.ts`. `coverImage` uses Astro's image helper (`ctx.image()`), so it must be a **relative path** to a local asset, not a remote URL.
- `src/pages/case-studies/[slug].astro` — dynamic route; `getStaticPaths` reads the collection.
- `src/assets/icons/`, `src/assets/images/` — SVGs and raster assets (imported via path aliases).
- `public/` — static passthrough (favicon, `robots.txt`, `images/`, `videos/`).
- `astro.config.mjs` — site URL is `https://fayda.studio`; integrations are `sitemap()` and `mdx()`.
- `tsconfig.json` — defines the path aliases (see below). Includes `.astro/types.d.ts` (generated by `astro dev`/`astro build`).

## Path aliases (use these, not relative imports)

Defined in `tsconfig.json` and resolved by Astro at build time. Grep for these before reaching for `../../`:

- `@components/*` → `src/components/*`
- `@chrome/*` → `src/components/chrome/*`
- `@ui/*` → `src/components/ui/*`
- `@sections/*` → `src/components/sections/*`
- `@layouts/*` → `src/layouts/*`
- `@data/*` → `src/data/*`
- `@assets/*` → `src/assets/*`
- `@images/*` → `src/assets/images/*`
- `@icons/*` → `src/assets/icons/*`

## Environment

`.env` is gitignored. The repo currently expects two public vars (consumed in `src/components/integration/posthog.astro`):

- `PUBLIC_POSTHOG_PROJECT_TOKEN`
- `PUBLIC_POSTHOG_HOST` (e.g. `https://eu.i.posthog.com`)

A starter `.env` is checked in for local dev. Do not commit a `.env.production` with real secrets.

## Conventions and gotchas

- **View transitions are load-bearing.** The Layout's `<ClientRouter />` plus the `__posthog_initialized` guard in `posthog.astro` are required to keep analytics working across soft navigation. Removing the guard causes a stack overflow on every page transition — do not refactor it out.
- **Lenis + view transitions:** the Lenis instance is re-bound on `astro:page-load` and reset on `astro:after-swap`. Any new global JS that touches scrolling must hook these events, not `DOMContentLoaded`.
- **One layout, no per-page SEO head:** title and description default to `siteData` and are overridden via `<Layout title=... description=...>`. Don't add a second `<head>` in pages.
- **Tailwind v4, not v3:** no `tailwind.config.*`; theme tokens (`--color-brand-*`, font families, etc.) are declared in `src/styles/global.css` inside an `@theme { ... }` block. Add new design tokens there.
- **Google site verification meta tag is hardcoded** in `Layout.astro` — leave it.
- **Generated, do not edit:** `.astro/` (Astro's generated types and content data store) and `dist/` (build output). Both are gitignored.
- **Typecheck flow:** `npm run type` runs `astro check`, which needs `.astro/types.d.ts` to exist. If it complains, run `npm run dev` (or `npm run build`) once to regenerate it.
- **No comments policy (per repo owner):** do not add code comments unless explicitly asked. Match the existing style of the file you're editing.

## Quick "did I break it?" checklist

1. `npm run type` — passes with no errors.
2. `npm run build` — completes; check that new pages appear under `dist/`.
3. Manually click through the affected route in `npm run dev` and watch the console — PostHog pageview events should fire and the navbar should still hide-on-scroll-down / show-on-scroll-up.
